# 장애 시나리오 테스트

## 시나리오 1: DB 장애
- 방법: postgres pod 강제 재시작
- 확인: `/readyz` 503 전환, 알람 발송, 자동 복구 후 정상 전환
- 기록: MTTD/MTTR, 실패 요청 수

## 시나리오 2: Worker 중단
- 방법: worker deployment scale 0
- 확인: queued 적체 증가, processing 완료 지연, 알람 발송
- 복구: scale 1 이상으로 복귀

## 시나리오 3: Web 장애
- 방법: web pod 연속 삭제
- 확인: Ingress를 통한 서비스 연속성, p95 변화, 5xx 비율

## 시나리오 4: 대용량 업로드 부하
- 방법: k6로 동시 업로드 요청
- 확인: 업로드 실패율, 처리 지연, CPU/메모리 사용량

- 복구 시간(MTTR)
- 데이터 무결성(영상/댓글/좋아요 손실 여부)
